01Princípios Inegociáveis

Antes de qualquer sprint começar, esses seis princípios precisam estar acordados entre todos os times. Eles não são valores — são restrições de design que moldam todas as decisões de arquitetura, produto e processo.

P-01
Plataforma, não Projeto
O frontend white label tem PM dedicado, OKRs próprios e roadmap contínuo. Não tem prazo de encerramento. Qualquer decisão que otimiza entrega de curto prazo às custas da extensibilidade futura é errada.
P-02
Contrato antes do Código
A interface CreditProductMFE, o schema do UCT, a Partner Config API e o payload do PartnerContext são definidos e versionados antes do primeiro sprint de produto. Quebrar contratos é mais caro que definir bem no início.
P-03
Times por Produto, não por Camada
Cada squad entrega end-to-end — da API ao MFE ao BFF. Nenhum ticket de feature atravessa mais de dois times. Se atravessa, o design organizacional está errado.
P-04
Segurança como Requisito de Entrada
UCT em memória, mTLS, revogação em tempo real e logs por 5 anos são inegociáveis. Pentest e threat modeling acontecem antes do go-live do primeiro parceiro, não depois. Isso não é negociável por prazo.
P-05
Documentação é Produto
Cada endpoint tem exemplos reais, não só schema. Cada erro tem código + causa + ação. Se dois parceiros fazem a mesma pergunta ao suporte, isso é um bug de documentação — não de suporte.
P-06
Primeiro Parceiro Testa a Plataforma
A pergunta não é "o parceiro ficou satisfeito?" — é "quanto do time central foi acionado para essa integração?". Se a resposta for "muito", você ainda não tem plataforma. Meça o tempo de onboarding desde o dia um.
✦ O princípio que não pode ser terceirizado

O modelo de autenticação (Federado ou Standalone) é uma decisão de segmentação de mercado, não de arquitetura. A plataforma suporta os dois. O que o Head decide é qual modelo cada segmento de parceiro recebe como padrão — e isso precisa estar explícito no contrato de onboarding antes de qualquer integração começar.

⚠ Regra de Ouro da Organização

Uma squad consegue fazer um deploy completo da sua feature sem aprovação ou trabalho de outra squad? Se não, o design organizacional está errado. A Squad 1 (Core Platform) é a única exceção justificada — ela provê infraestrutura para todas. Por isso ela precisa ter capacidade proporcional ao consumo das demais.

06Contratos Técnicos

Definidos no Sprint Zero, antes do primeiro commit de produto. Quebrar esses contratos é mais caro que definir bem no início.

TypeScript · Contratos da Plataforma
// ── CONTRATO 1: Todo MFE de produto deve implementar ────────────
interface CreditProductMFE {
  productId:     string;          // "capital-giro" | "antecipacao-recebiveis"
  version:       string;          // semver obrigatório
  capabilities:  Capability[];    // simulate | apply | track | renegotiate

  mount(el: HTMLElement, ctx: PartnerContext): void;
  unmount(): void;
  onSessionExpired(): void;
  onThemeChange(theme: PartnerTheme): void;
}

// ── CONTRATO 2: PartnerContext injetado pelo Shell ───────────────
interface PartnerContext {
  partnerId:     string;          // "banco-xpto" | "vivo" | "claro"
  partnerTenant: string;          // isolamento multi-tenant
  theme:         PartnerTheme;    // tokens de design resolvidos
  accessToken:   string;          // UCT — apenas em memória
  sessionId:     string;
  correlationId: string;          // rastreabilidade ponta a ponta
  authMode:      'federated' | 'standalone';
}

// ── CONTRATO 3: Partner Config API response ──────────────────────
interface PartnerConfig {
  partnerId:    string;
  brand:        PartnerBrand;     // cores, fonte, logo, favicon
  products:     string[];         // produtos habilitados
  features:     FeatureFlags;    // flags por parceiro
  authMode:     'federated' | 'standalone';
  sessionConfig: {
    timeoutMs:  number;           // inatividade em ms
    mfaRequired: boolean;         // obrigatório no Standalone
  };
}
Política de Versionamento

Contratos usam semver. Breaking changes (remoção de campo, mudança de tipo) exigem 90 dias de aviso e migration guide publicado no Portal do Desenvolvedor antes de entrar em vigor. Additive changes (novos campos opcionais) não exigem versão nova.

08Governança

Chapters horizontais garantem padrão técnico entre squads. Rituais claros evitam que decisões importantes fiquem nos corredores.

Chapters Horizontais
🔐
Chapter de Segurança
Pentest semestral, threat modeling de novos produtos, revisão de controles antes de qualquer go-live. Reporta para Cibersegurança Bradesco.
🏗️

ADRs obrigatórios para decisões estruturais. Revisão cruzada antes de nova capacidade de crédito. Golden path documentado e enforced no CI/CD.

Fases do roadmap revisadas. Ajuste de prioridades entre squads. Head de Tecnologia + PMs. Inclui revisão do DX Score.
⚠ Governança Aarin × Bradesco

Desenvolvimento na Aarin exige SLA intragrupo formal com penalidades reais. PI (Propriedade Intelectual) 100% Bradesco em contrato. Pentest e aprovações de segurança sempre pela área Bradesco — nunca pela Aarin. Dados de clientes e ambientes de produção sempre no perímetro Bradesco/Azure. A Aarin opera como fábrica de software do grupo, não como detentora do produto.

09Plano de Rollout por Parceiro

O rollout de parceiros segue quatro estágios deliberados. Cada estágio valida premissas do próximo — pular estágios é o erro mais comum em plataformas financeiras.

Estágio 1 · Fase 1
Piloto Interno — Aarin
Valida a plataforma sem pressão externa. Ambiente controlado, dados sintéticos, time técnico interno. Meta: descobrir todos os problemas antes que um parceiro real os encontre. Duração: 4–6 semanas.
Estágio 2 · Fase 2
Parceiro Âncora
Um parceiro estratégico com alto nível técnico co-constrói com o time. Acesso antecipado em troca de feedback estruturado. Suporte dedicado. Meta: validar o playbook de onboarding e medir tempo real de integração. Duração: 6–8 semanas.
Estágio 3 · Fase 3
Abertura Controlada · 3–5 Parceiros
Com o playbook validado, abre para um grupo pequeno e diverso de parceiros. Monitoramento intenso de métricas de onboarding e suporte. Cada novo parceiro testa uma hipótese de go-to-market. Duração: 1 trimestre.
Estágio 4 · Fase 4
Self-Service Completo
Parceiro acessa o portal, configura o tema, habilita produtos e recebe credenciais sem nenhuma interação humana do time Bradesco. Esse é o critério de maturidade de plataforma. Meta: onboarding em <5 dias sem ticket de suporte.
✦ Critério de Maturidade — a pergunta que define tudo

Em cada estágio, a pergunta a fazer após cada integração é: "quanto do time central foi acionado para que esse parceiro fosse ao ar?" Se a resposta for "muito", você ainda não tem plataforma — tem um projeto customizado com visual branco. A redução sistemática dessa fricção, medida sprint a sprint, é o indicador mais honesto de que a estratégia está funcionando.
