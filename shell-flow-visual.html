Fase 1 — Shell inicializa
01
S-01 SHELL
Parceiro embute o Shell no app
O app do Banco XPTO carrega o Shell via <iframe> ou micro frontend host, passando o partnerId como parâmetro inicial.
// App do parceiro <ShellApp partnerId="banco-xpto" onEvent={handleShellEvent} />
✓ Shell montado no DOM
02
S-02 TEMA
Busca configuração do parceiro
Shell faz GET na Partner Config API usando o partnerId. Recebe cores, tipografia, logo, favicon e quais produtos estão ativos.
GET /v1/partners/banco-xpto/config → { primaryColor: "#0055A4", logo: "cdn.xpto.com/logo.svg", products: ["capital-giro"] }
✓ Config carregada em 42ms
03
S-03 TEMA
Aplica tema em runtime
Shell injeta os Design Tokens como CSS Custom Properties no :root. Toda a tela muda de aparência instantaneamente — zero rebuild.
// Injeção runtime — sem rebuild root.setProperty('--color-primary', '#0055A4') root.setProperty('--font-family', 'Nunito') document.title = 'Crédito XPTO'
✓ Identidade visual aplicada
Fase 2 — Autenticação B2B2X
04
S-04 AUTH B2B2X
Parceiro autentica o cliente (X)
O Banco XPTO usa seus próprios sistemas (biometria, senha) para autenticar o cliente final. O Bradesco não vê esse fluxo. O resultado é o Partner User Token.
// Acontece no sistema do parceiro const partnerToken = await xptoAuth.authenticateUser(biometria) // → token do usuário no sistema XPTO
✓ Cliente autenticado no parceiro
05
S-05 AUTH B2B2X
Token Exchange — emissão do UCT
Parceiro chama o endpoint de Token Exchange (RFC 8693). A plataforma valida o PAT + Partner User Token e emite o UCT — que identifica X dentro do tenant do parceiro B.
POST /oauth2/token/exchange Authorization: Bearer {PAT} subject_token: {partnerUserToken} → { access_token: "UCT...", expires_in: 900 }
✓ UCT emitido · válido 15min
06
S-06 AUTH B2B2X
Shell armazena UCT em memória
O Shell recebe o UCT via postMessage do parceiro e o mantém somente em memória JavaScript — nunca em localStorage ou cookie acessível ao JS. XSS não consegue exfiltrá-lo.
// NUNCA localStorage — apenas memória let uct = null; // variável em closure window.addEventListener('message', (e) => { if (e.data.type === 'auth:token') uct = e.data.uct; // salvo em memória })
✓ UCT em memória · seguro
Fase 3 — Carregamento do MFE
07
S-07 MFE REGISTRY
Descobre versão ativa do produto
Shell consulta o MFE Registry para saber qual versão do módulo "capital-giro" está ativa em produção. Nunca hardcoda URLs.
GET /v1/mfe/registry → { "capital-giro": "2.4.1" } // URL montada dinamicamente mfe.baas.bradesco.com.br/ capital-giro/2.4.1/remoteEntry.js
✓ Versão 2.4.1 localizada
08
S-08 MODULE FEDERATION
Carrega o módulo remoto via CDN
Webpack Module Federation baixa o remoteEntry.js do CloudFront. O MFE é carregado em isolamento — sem acesso ao estado interno do Shell ou de outros MFEs.
// Module Federation carrega em runtime const CapGiro = await loadRemote( 'capitalGiro/CapitalGiroApp' ) // → bundle do MFE via CloudFront
✓ MFE baixado em 87ms
09
S-09 PRODUTO ATIVO
Shell monta o MFE com contexto completo
Shell chama mfe.mount() passando o PartnerContext — tema, UCT, partnerId e correlationId. O produto renderiza com a identidade do parceiro. Usuário vê a jornada de crédito.
CapGiro.mount(container, { partnerId: 'banco-xpto', theme: resolvedTheme, accessToken: uct, // UCT da memória correlationId: 'req_9a1c', })
🎉 Jornada de crédito ativa!
