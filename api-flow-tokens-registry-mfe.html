00Arquitetura & Cadeia de Chamadas

Três fluxos independentes mas encadeados. O Fluxo 1 ocorre uma vez no bootstrap (identidade do parceiro). O Fluxo 2 ocorre no início de cada sessão (identidade do cliente). O Fluxo 3 ocorre em cada interação do usuário com o MFE.

Fluxo 1 · Bootstrap
Tokens do Parceiro
Ocorre uma vez por sessão de parceiro. O PAT autentica o B, o Token Exchange gera o UCT para o X. Os design tokens chegam junto com a config.
Fluxo 2 · Session
Customer Registry
Consulta CPF/CNPJ para verificar cadastro, limite pré-aprovado e rating. Alimenta o hero do B2C e o banner do embedded antes do usuário interagir.
Fluxo 3 · Produto
APIs de MFE
Cada interação do usuário (simular, aplicar, consultar) passa pelo BFF que valida o UCT, troca por Service Token e agrega as respostas das APIs de LaaS/CaaS.
⚡ Regra de ouro: o MFE nunca chama APIs de plataforma diretamente

Todo tráfego do MFE passa pelo BFF. O BFF é a única camada que detém credenciais de serviço (mTLS + Service Token). O browser nunca tem acesso a credenciais de backend.

O backend do parceiro deve implementar renovação proativa: quando o PAT tiver menos de 3 minutos para expirar, iniciar novo request de client_credentials antes de precisar. Nunca deixar o PAT expirar em produção.

04Customer Registry

O Customer Registry é o repositório central de identidade dos clientes finais (X) dentro da plataforma BaaS. Ele é consultado no bootstrap da sessão para alimentar o hero do B2C, o banner do embedded, e verificar se o cliente já tem cadastro ativo ou precisa ser criado.

Dois momentos de consulta ao Registry

Momento 1 — Bootstrap da sessão: BFF consulta o registry com o document_hash do UCT para trazer limit pré-aprovado e rating. Alimenta a UI antes do usuário interagir.
Momento 2 — Originação: ao aplicar para crédito, o Core Platform verifica o registry novamente para dados atualizados e cria/atualiza o registro da operação.

GET /v1/customers/{documentHash} Consulta cliente por hash do documento · chamado pelo BFF
BFF → CoreCache: 60s

O BFF nunca envia o CPF/CNPJ raw para a API — sempre o hash SHA-256 do documento. O Core Platform faz o lookup internamente. Isso reduz a superfície de exposição de PII no tráfego inter-serviço.

⚠ Idempotência obrigatória

O header X-Idempotency-Key é obrigatório. Use o simulationId como chave. Se a request for repetida com a mesma chave dentro de 24h, o servidor retorna o resultado original sem criar nova operação. Protege contra duplo-clique do usuário e retries de rede.

async function callLaaS(endpoint, payload) {
  return circuitBreaker.execute(async () => {
    let attempt = 0;
    while (attempt < 3) {
      try {
        const res = await fetch(endpoint, { /* mTLS + ServiceToken */ });
        if (res.ok) return res.json();
        const err = await res.json();
        if (!err.retryable) throw new NonRetryableError(err);
      } catch (e) {
        attempt++;
        if (attempt === 3) throw e;
        await sleep(1000 * Math.pow(2, attempt)); // 1s, 2s, 4s
      }
    }
  });
}
10Segurança — Resumo dos Controles

Resumo dos controles de segurança por camada. Cada camada tem responsabilidade específica — nenhum controle é delegado para cima ou para baixo da cadeia.

Browser (MFE)
Responsabilidades
UCT apenas em memória JS. Refresh Token em HttpOnly Cookie. PKCE em todo authorization_code flow. Sem credenciais de serviço. CSP Header enforced.
BFF
Responsabilidades
Validação de UCT (assinatura + blacklist + partner_match). Token Exchange para Service Token via mTLS. Rate limit por usuário. Sanitização de inputs. Log de toda operação.
Auth Server
Responsabilidades
Emissão e revogação de PAT, UCT e Service Token. Validação de certificados mTLS. Token blacklist no Redis. Rastreabilidade via jti. Rotação de chaves JWKS.
Core Platform
Responsabilidades
Aceita apenas Service Token (nunca UCT direto). Isolamento multi-tenant via X-Tenant header. Rate limit global por parceiro. Auditoria de todas as operações financeiras (retenção 5 anos).
🔒 Controles inegociáveis

UCT em memória · nunca localStorage ou sessionStorage
mTLS obrigatório · para qualquer chamada BFF → Core Platform ou Auth Server
Document hash · CPF/CNPJ nunca trafega raw entre serviços — apenas SHA-256
Idempotency Key · obrigatório em todos os endpoints de escrita
Audit log · toda operação de crédito armazenada por mínimo 5 anos (Bacen Res. 4.658)
