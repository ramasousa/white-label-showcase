00 Contexto & Decisões Principais

Este documento consolida as decisões arquiteturais para a plataforma de crédito white label do BaaS, cobrindo os quatro produtos iniciais e o modelo de evolução pós-entrega. O princípio fundamental: o frontend white label é um produto de plataforma, não um projeto de UI.

💳
Produtos de Crédito
Capital de Giro, Crédito Emergencial, Mais Prazo e Antecipação de Recebíveis — cada um como um Micro Frontend independente e versionado.
🏗️
Modelo B2B2X
Bradesco (plataforma) → Parceiro B (distribui) → Cliente X (contrata crédito). Dois pares de confiança distintos, com autenticação delegada por Token Exchange.
🎨
White Label por Parceiro
Identidade visual, logo, favicon, cores, tipografia e fluxos configuráveis por parceiro via Portal — sem rebuild de código.
🔌
Extensibilidade
Novos produtos são adicionados como módulos remotos no Module Federation — o Shell descobre produtos via API, sem hardcode e sem redesploy.
01 Stack Tecnológica & Arquitetura MFE

React é a escolha correta e está alinhado com o que a Squad 4 (Experience & Ecosystem) já definiu para os Micro Frontends. A arquitetura usa Module Federation para isolamento por produto.

✔ Decisão: React 18 + TypeScript

Ecossistema de theming runtime mais maduro (CSS Custom Properties + Emotion), Module Federation estabelecido, maior pool de talentos no mercado de fintechs, biblioteca de componentes acessíveis (Radix UI). Angular seria justificado apenas se o parceiro estratégico principal exigisse code-sharing com sua base Angular existente.

Como os Tokens São Injetados (Runtime)

TypeScript · Shell App · applyTheme.ts 
// Shell carrega tema ao inicializar — zero rebuild
async function applyPartnerTheme(partnerId: string) {
  const config = await fetch(`/v1/partners/${partnerId}/config`);
  const { brand } = await config.json();

  const root = document.documentElement;

  // Injeta como CSS Custom Properties — herança automática em todos os MFEs
  root.style.setProperty('--color-primary',    brand.primaryColor);
  root.style.setProperty('--color-secondary',  brand.secondaryColor);
  root.style.setProperty('--color-surface',    brand.surfaceColor);
  root.style.setProperty('--font-family',      brand.fontFamily);
  root.style.setProperty('--border-radius',    brand.borderRadius);
  root.style.setProperty('--shadow-elevation', brand.shadowElevation);

  // Carrega fonte do parceiro dinamicamente
  if (brand.fontUrl) {
    const link = document.createElement('link');
    link.rel = 'stylesheet';
    link.href = brand.fontUrl;
    document.head.appendChild(link);
  }

  // Favicon
  document.querySelector('link[rel="icon"]').href = brand.faviconUrl;
  document.title = brand.appName;
}

Camada 2 — Configuração de Produtos

Configuração de quais produtos estão ativos, limites de crédito expostos, campos obrigatórios, copy de textos legais e fluxo de formalização digital. Controlada por contrato comercial com o parceiro, editável via Portal com aprovação.

Camada 3 — Customização Enterprise (via contrato)

Para parceiros estratégicos: troca de componentes via slot/composition pattern, steps adicionais na jornada (ex: validação de dados proprietária do parceiro), copywriting completo customizado e integração de sistemas de score do próprio parceiro.

⚠ Regra de Ouro do Design System

O Design System interno (componentes base) nunca expõe classes CSS ou DOM interno. Toda customização é via tokens. Isso garante que o Bradesco possa evoluir os componentes sem quebrar os temas dos parceiros.

03 Autenticação B2B2X

O modelo tem dois pares de confiança distintos e isso é a chave para entender toda a arquitetura de auth. O cliente X não tem relação direta com o Bradesco — ele é cliente do parceiro.

🏦
Par 1: Bradesco ↔ Parceiro (B)
Confiança de plataforma. OAuth2 Client Credentials + mTLS. O parceiro tem certificado emitido pelo Bradesco. Token de 15 min, renovação automática. Escopo restrito ao partnerId e produtos contratados.
👤
Par 2: Parceiro (B) ↔ Cliente (X)
Confiança de produto. O parceiro é o IdP do usuário final. Pode usar biometria, senha, OpenID Connect próprio. O resultado é um token do parceiro que será trocado pelo UCT via Token Exchange.
STEP 1

Parceiro autentica com a plataforma (M2M)

Usando client_credentials + mTLS, o parceiro obtém o Partner Access Token (PAT). TTL: 15 minutos. Escopo: produtos contratados + permissão de token exchange. Renovação automática via background refresh antes da expiração.

STEP 2

Parceiro autentica seu cliente final (X) com os sistemas próprios

O parceiro usa a stack de autenticação dele — pode ser biometria do app mobile, login com senha, OpenID Connect interno. O Bradesco não vê esse fluxo. O resultado é um Partner User Token que identifica o X no contexto do parceiro.

STEP 3

Token Exchange (RFC 8693)

O parceiro chama o endpoint de Token Exchange passando o PAT (autenticação M2M) e o Partner User Token (identidade do X). A plataforma valida o PAT, verifica se o parceiro tem permissão de exchange, e emite o User Context Token (UCT).

STEP 4

UCT é usado pelo Shell e pelos MFEs

O Shell recebe o UCT e o mantém apenas em memória (nunca em localStorage). Todos os MFEs consomem o UCT via PartnerContext. O UCT identifica X dentro do tenant do parceiro B, com os escopos de crédito autorizados.

Estratégia de Storage por Token

Token Storage Justificativa TTL
UCT (User Context Token) Memória JS apenas XSS não consegue exfiltrar — não persiste entre tabs 15 min
Refresh Token HttpOnly Cookie SameSite=Strict + Secure + Path=/auth — inacessível ao JS 8h
PAT (Partner Access Token) Backend do parceiro Nunca chega ao frontend — gerenciado server-side pelo parceiro 15 min
PKCE code_verifier sessionStorage Válido apenas durante o fluxo de auth, descartado após Transient
localStorage / cookies não-HttpOnly PROIBIDO Vulnerável a XSS — violação de política de segurança —

Revogação de Tokens

Cenários de Revogação
Logout voluntário do X
  └─ Shell chama DELETE /v1/sessions/{session_id}
     └─ Core Platform: jti → Redis blacklist (TTL = exp do UCT)
     └─ Cookie de Refresh Token invalidado com Max-Age=0

Fraude detectada pelo motor de risco
  └─ Evento Kafka: user.session.revoke com { partner_id, sub, jti }
     └─ Auth service processa em <200ms → jti no blacklist Redis
     └─ Shell recebe evento via SSE → session:revoked → desmonta MFEs

Parceiro revoga acesso de usuário (ex: cliente encerrou conta)
  └─ Parceiro chama POST /v1/partners/{id}/users/{sub}/revoke
     └─ Revogação imediata — todos os UCTs ativos daquele sub invalidados
05 Controle de Sessão & Lifecycle

O Shell é o único responsável pelo ciclo de vida da sessão. Os MFEs são stateless em relação à autenticação — recebem o UCT via contexto e reagem a eventos de sessão.

Fluxo de Sessão Completo
Usuário acessa o MFE via parceiro └─ Shell verifica UCT em memória ├─ Válido → injeta no PartnerContext → MFE renderiza │ ├─ Expirado → Silent Refresh │ └─ fetch POST /oauth2/token com Refresh Token Cookie (HttpOnly) │ ├─ Sucesso → novo UCT em memória → continua │ └─ Falha → dispara evento session:refresh-failed │ └─ Shell chama parceiro via postMessage("auth:required") │ └─ Parceiro reautentica X e repassa novo UCT │ └─ Ausente → redireciona para fluxo de auth do parceiro

Timeout de Inatividade

TypeScript · Shell · inactivity-monitor.ts 
// Timeout configurável por parceiro via PartnerConfig
const INACTIVITY_TIMEOUT = partnerConfig.sessionTimeoutMs ?? 600_000; // 10min default
const WARN_BEFORE_MS = 120_000; // avisa 2min antes

let timer: ReturnType<typeof setTimeout>;

function resetTimer() {
  clearTimeout(timer);
  timer = setTimeout(() => showExpiryWarning(), INACTIVITY_TIMEOUT - WARN_BEFORE_MS);
}

// Monitora eventos de atividade do usuário
['mousemove', 'keydown', 'click', 'touchstart', 'scroll']
  .forEach(evt => window.addEventListener(evt, resetTimer, { passive: true }));

function showExpiryWarning() {
  // Emite evento para o MFE ativo mostrar modal de aviso
  window.dispatchEvent(new CustomEvent('session:expiry-warning', {
    detail: { expiresInMs: WARN_BEFORE_MS }
  }));

  // Se usuário não interagir, encerra sessão
  setTimeout(() => terminateSession('inactivity'), WARN_BEFORE_MS);
}

Pipeline de CI/CD por MFE

CI/1

Build & Testes Automatizados

vitest (unitários) + playwright (e2e no sandbox) + chromatic (regressão visual). Qualquer falha bloqueia o pipeline. Coverage mínimo: 80% nas jornadas críticas.

CI/2

SAST & Dependências

Snyk para vulnerabilidades de dependências + SonarQube para análise estática de segurança. Violações críticas bloqueiam o pipeline — não podem ser suprimidas sem aprovação do Chapter de Segurança Bradesco.

CD/1

Deploy em Staging

Artefato publicado em S3 com prefixo de versão. CloudFront aponta para staging. Smoke tests automatizados executam com parceiro de teste. Revisão manual de UX pelo PM da Squad 4.

CD/2

Canary Release em Produção

Nova versão liberada para 5% do tráfego via CloudFront weighted routing. Monitoramento de error rate, LCP e CLS por 30 minutos. Se métricas OK → progressão para 100%. Se anomalia → rollback automático em <2 minutos.

CD/3

Atualização do MFE Registry

Após 100% do tráfego na nova versão, a API de registry é atualizada. Novos loads do Shell já recebem a nova versão. Versões anteriores permanecem no S3 por 90 dias para rollback manual.

07 AWS vs Azure — Decisão de Cloud

Esta é uma decisão estratégica com impacto de longo prazo. A resposta não é "qual cloud é melhor" — é "onde cada camada deve rodar considerando governança, compliance bancário e custo operacional".

Dimensão Camada de Frontend (MFEs / Shell) Camada de APIs (Core Platform / BaaS)
Cloud Natural AWS (Aarin já usa) Azure (Bradesco já usa)
Serviços-chave S3 + CloudFront + Route 53 Azure APIM + AKS + Azure AD B2C
Dados de clientes Não trafega dados sensíveis no CDN Azure — dentro do perímetro regulatório Bradesco
Compliance Bacen Aplica-se parcialmente (artefatos estáticos) Azure — auditoria e controles já mapeados
Latência AWS CloudFront (melhor CDN global) Azure — latência API não é gargalo vs CDN
Custo operacional Baixo — artefatos estáticos são baratos Azure — já pago como parte do contrato Bradesco
Time operando hoje Aarin tem proficiência em AWS Bradesco TI tem proficiência em Azure
✦ Decisão Recomendada

Arquitetura Híbrida: AWS para Frontend, Azure para APIs

O frontend (Shell App + MFEs) roda em AWS CloudFront + S3 — Aarin tem proficiência, custo baixo e melhor CDN do mercado para artefatos estáticos. As APIs do Core Platform, LaaS e CaaS ficam na Azure, dentro do perímetro regulatório e de compliance do Bradesco. A comunicação é via HTTPS puro — não há acoplamento de infraestrutura. Esta separação também garante que dados de clientes nunca passem pela AWS.

⚠ Regra Inegociável: Dados de Clientes Ficam no Azure

O CloudFront/S3 serve apenas artefatos estáticos (JS, CSS, imagens dos MFEs). Nenhum dado de cliente X trafega ou é armazenado na AWS. As chamadas de API dos MFEs vão diretamente para os endpoints no Azure via HTTPS. Esta separação deve ser auditada pela área de Segurança e Compliance do Bradesco antes do go-live.

Mapa de Serviços por Cloud

🟠 AWS (Aarin)
S3 — artefatos MFE versionados CloudFront — CDN global Route 53 — DNS WAF — proteção da camada de CDN CodePipeline — CI/CD dos MFEs
🔵 Azure (Bradesco)
Azure APIM — API Gateway AKS — microsserviços Azure AD B2C — IAM / OAuth2 Azure Redis — token blacklist Azure Event Hub — Kafka Azure Monitor — observabilidade
🟢 Fronteira Segura
HTTPS apenas CORS restrito a domínios parceiros whitelist Zero dados de cliente na AWS CSP Headers nos artefatos
08 Aarin como Desenvolvedora Intragrupo

Desenvolver na Aarin é tecnicamente correto — ela já tem DX Score de 78/100 e maturidade em produtos digitais. Mas exige governança clara para evitar que "intragrupo" vire "sem cobrança".

✅
Vantagens da Aarin
Maturidade em desenvolvimento de produto digital, AWS já operacional, processos ágeis, contratação de perfis tech sem as travas do RH bancário, DX Score superior ao Bradesco hoje.
⚠️
Riscos a Gerenciar
Tratamento como fornecedor cativo sem SLAs reais, dados de clientes saindo do perímetro regulatório do Bradesco, padrões de segurança distintos, PI (Propriedade Intelectual) mal definida.

Modelo de Governança Recomendado

GOV 1

SLA Intragrupo Formal com Penalidades Reais

Uptime: 99.9% para o Shell, 99.5% para MFEs individuais. Tempo de deploy de novo MFE: máx. 5 dias úteis da aprovação. Incidentes P1: resposta em 15 minutos, resolução em 2h. Sem penalidade real, o SLA é decorativo.

GOV 2

Pentest e Segurança pela Área Bradesco

SAST/DAST no pipeline da Aarin, mas revisão de pentest e threat modeling pela área de Cibersegurança do Bradesco — não pela Aarin. O desenvolvimento é intragrupo, o perímetro de risco é do banco.

GOV 3

Propriedade Intelectual 100% Bradesco

Todo código produzido, Design System, contratos de API e design tokens pertencem ao Bradesco. A Aarin opera como fábrica de software do grupo, não como detentora do produto. Contrato intragrupo deve ser explícito nisso.

GOV 4

Dados de Clientes: Zero na Aarin

Ambiente de desenvolvimento da Aarin usa dados sintéticos (faker + gerador de CPF de teste). Nenhum ambiente Aarin (dev, staging) toca dados reais de clientes X. Produção roda em infraestrutura do perímetro Bradesco.

09 Adição de Novos Produtos Pós-Entrega

Este é o teste definitivo de uma boa arquitetura de plataforma: o custo marginal de adicionar o 5º produto deve ser dramaticamente menor que o do 1º. Com Module Federation + MFE Registry, isso é garantido por design.

O Que Muda (e o Que Não Muda)

Componente Novo Produto Requer Alteração? Detalhe
Shell App Não — zero alteração Descobre produtos via API registry dinamicamente
MFEs existentes Não — isolados por design Module Federation garante isolamento total
Design Tokens / Tema Não — herança automática CSS Custom Properties do :root se propagam

Princípios que Guiam Todas as Decisões

🔒
Security by Design
UCT apenas em memória. Refresh Token em HttpOnly Cookie. Zero dados de clientes na AWS. Revogação em tempo real via Redis. Pentest pela área de Cibersegurança Bradesco.
🧩
Modularidade Real
MFEs independentes com contrato tipado. Shell descobre produtos via API. Novos produtos sem redesploy do Shell. Versões anteriores mantidas por 90 dias para rollback.
🎨
Theming Runtime
CSS Custom Properties propagados via Shell. Nenhum rebuild para trocar identidade de parceiro. Design Tokens como API — não como código.
📏
Contratos Explícitos
Interface CreditProductMFE obrigatória. API-first entre squads. Versioning semântico nos MFEs. Breaking changes com 90 dias de aviso e migration guide.
☁️
Híbrido com Propósito
AWS para o frontend estático (Aarin, custo baixo, melhor CDN). Azure para APIs e dados (Bradesco, compliance, perímetro regulatório). Não por ideologia, por adequação.
🏭
Plataforma, não Projeto
O frontend white label tem OKRs, NPS de parceiros, roadmap próprio e PM dedicado. Não é entregável com prazo de encerramento — é produto contínuo.
