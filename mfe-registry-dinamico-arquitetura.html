Arquitetura · Shell App Module Federation Zero-Deploy Product Ops

Adicionar ou remover
um MFE sem deploy
do Shell App.

Como desacoplar a decisão de portfólio de produto (o que está disponível, para quem) do ciclo de deploy de engenharia — usando um MFE Registry dinâmico carregado em runtime via Module Federation.

A peça central
O Shell App nunca importa um MFE pelo nome em tempo de build. Ele lê um manifesto JSON em runtime que diz onde cada módulo remoto está hospedado — e esse manifesto é editado por um time de operações, não por um time de engenharia.
01
O problema: acoplamento entre produto e deploy

Sem essa arquitetura, toda mudança de portfólio — adicionar CDC, desabilitar Antecipação para um parceiro específico — exige uma alteração de código no Shell App, revisão, build, QA e deploy. Isso significa que o time de produto está sempre na fila do time de plataforma.

O que continua exigindo deploy do Shell

Mudanças na casca — autenticação, roteamento base, Design System core, contrato `CreditProductMFE`, ou a própria lógica de carregamento dinâmico. O Shell continua tendo ciclo de release próprio; ele só para de ser o gargalo para decisões de portfólio de produto.

02
Como o carregamento dinâmico funciona

Module Federation (Webpack 5) suporta dois modos de consumo de módulos remotos: estático (declarado em build-time no webpack.config.js) e dinâmico (resolvido em runtime via uma função que busca a URL do remote). A arquitetura do BaaS usa o modo dinâmico exclusivamente.

04
Como o Shell consome isso (código real)

O padrão é "dynamic remote container" do Webpack 5. Em vez de declarar os remotes no ModuleFederationPlugin, o Shell registra os remotes em runtime, lendo a URL do Partner Config carregado no bootstrap.

TypeScript · Shell · loadRemoteModule.ts
// Esta função é TUDO que o Shell precisa saber em build-time.
// Ela nunca menciona "capital-giro", "cdc" ou qualquer produto específico.

async function loadRemoteModule(remoteUrl: string, remoteName: string, exposedModule: string) {
  // 1. Injeta a tag <script> do remoteEntry.js dinamicamente
  await loadScript(remoteUrl);

  // 2. Acessa o container global exposto pelo MFE remoto
  const container = (window as any)[remoteName];
  await container.init(__webpack_share_scopes__.default);

  // 3. Resolve o módulo exposto (ex: "./CreditProductMFE")
  const factory = await container.get(exposedModule);
  return factory();
}

// No Shell, ao montar a navegação:
async function mountProductMFEs(partnerConfig: PartnerConfig) {
  const registry = await fetchMFERegistry();  // catálogo global

  for (const productId of partnerConfig.products) {
    const mod = registry.modules.find(m => m.id === productId && m.status === 'active');
    if (!mod) continue;  // produto não existe ou foi desabilitado — Shell ignora

    const CreditProductMFE = await loadRemoteModule(
      mod.remoteUrl, mod.remoteName, mod.exposedModule
    );

    registerProductTab({
      id: mod.id,
      displayName: mod.displayName,
      icon: mod.icon,
      component: CreditProductMFE,  // contrato comum a todos os MFEs
    });
  }
}
Por que isso é seguro

O contrato CreditProductMFE é tipado e versionado (contractVersion). O Shell só monta um MFE remoto se a versão do contrato for compatível com a que o Shell sabe consumir. Se o time de CDC mudar o contrato de forma incompatível, o Shell detecta no contractVersion e recusa montar — falha de forma segura, não quebra a aplicação.

05
Quem pode editar o quê — separação de responsabilidades
🛠️
Time de Produto (ex: CDC)
Builda e deploya o próprio bundle MFE de forma independente — CI/CD próprio, sem depender do Shell. Notifica o time de operações que o módulo está pronto para cadastro.
👤
Funcionário Bradesco (Operações)
Cadastra o módulo no MFE Registry via Portal Administrativo. Decide quais parceiros têm acesso a cada produto. Não escreve nem revisa código — apenas configura.
🏗️
Time de Shell App
Mantém o mecanismo de carregamento dinâmico, o contrato CreditProductMFE e a infraestrutura central. Faz deploy apenas quando a casca muda — não quando o portfólio muda.
Checklist — antes de habilitar um MFE em produção via Portal Administrativo
✓O bundle do MFE passou pelo health check automatizado (healthCheckUrl responde 200)
✓O contractVersion declarado é compatível com a versão suportada pelo Shell atual
✓O bundle foi testado em ambiente de staging com pelo menos um parceiro piloto
✓Design Tokens do(s) parceiro(s) alvo foram validados visualmente com o novo MFE
✓Rollback definido: alterar status para disabled remove o módulo do ar em segundos, sem deploy
⚡ A pergunta que isso deveria gerar internamente

Se o Shell nunca precisa de deploy para mudanças de portfólio, qual é o cadência de release que resta para o time de Shell App? A resposta correta é: evolução do mecanismo de federation, do Design System core, e do contrato de MFE — não mais "adicionar produto X". Isso deveria reduzir drasticamente a fila de deploy do time de plataforma e é um dos argumentos mais fortes para justificar o investimento na arquitetura para a liderança.
