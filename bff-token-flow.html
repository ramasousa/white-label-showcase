Visão em Camadas com BFF

Cada camada fala um protocolo diferente com a camada abaixo. O BFF é a fronteira onde o mundo do browser (HTTPS + UCT) se traduz para o mundo das APIs bancárias (mTLS + credenciais server-side).

Fluxo Completo de Tokens com BFF

Do clique do usuário no MFE até a resposta da API de crédito — com cada token, cada validação e cada troca de protocolo explicitada.

01
MFE → BFF REQUISIÇÃO
MFE chama o BFF com o UCT no header
O usuário clica em "Simular Capital de Giro". O MFE faz uma chamada HTTP para o BFF. O UCT (que estava em memória no Shell, repassado via PartnerContext) vai no header Authorization.
HTTP · MFE → BFF
GET https://bff-capital-giro.baas.bradesco.com.br/v1/simulate
Authorization: Bearer {UCT}          ← token em memória do Shell
X-Partner-Id: banco-xpto
X-Correlation-Id: req_9a1c-f3b2
X-Session-Id: sess_4d2e-8a71
Content-Type: application/json

{ "amount": 50000, "term": 12 }   ← dados da simulação
Por que o MFE não chama a API diretamente?

Porque a API de plataforma exige mTLS com certificado server-side — o browser não tem como fazer isso com segurança. O BFF é a única camada que detém as credenciais de serviço.

02
BFF · validação SEGURANÇA
BFF valida o UCT — 3 verificações
Antes de qualquer outra coisa, o BFF valida o UCT recebido. Se qualquer verificação falhar, retorna 401 e não prossegue.
Node.js · BFF · validateUCT.ts
async function validateUCT(uct: string, req: Request) {

  // ① Verifica assinatura JWT (chave pública do Auth Server)
  const payload = await jwtVerify(uct, JWKS_URL);

  // ② Verifica se o jti não está na blacklist (Redis)
  const revoked = await redis.get(`blacklist:${payload.jti}`);
  if (revoked) throw new Error('TOKEN_REVOKED');

  // ③ Verifica se o parceiro do token == parceiro da request
  if (payload.partner_id !== req.headers['x-partner-id'])
    throw new Error('PARTNER_MISMATCH');

  return payload; // { sub, partner_id, scope, products, exp }
}
03
BFF → Core Platform TOKEN EXCHANGE
BFF troca UCT por Service Token via mTLS
Com o UCT validado, o BFF chama o Authorization Server para obter um Service Token — uma credencial server-to-server que identifica tanto o serviço (BFF) quanto o contexto do usuário (X) e do parceiro (B).
HTTP · BFF → Auth Server · mTLS
# BFF faz isso com certificado mTLS do lado do cliente
POST https://auth.baas.bradesco.com.br/oauth2/token
# [certificado TLS do BFF apresentado automaticamente]

grant_type=urn:ietf:params:oauth:grant-type:token-exchange
&subject_token={UCT}
&subject_token_type=urn:ietf:params:oauth:token-type:access_token
&requested_token_type=urn:ietf:params:oauth:token-type:access_token
&scope=credit.capital-giro:simulate credit.capital-giro:apply
&audience=lending-api.internal.bradesco.com.br

── RESPOSTA ──────────────────────────────────
{
  "access_token": "{ServiceToken}",   ← token server-to-server
  "token_type":   "Bearer",
  "expires_in":   300                  ← 5 min, escopo restrito
}
O que o Service Token carrega que o UCT não tem

O Service Token é emitido com audience específico (lending-api), TTL muito menor (5 min vs 15 min do UCT) e inclui a identidade do BFF como emissor delegado. As APIs de plataforma só aceitam Service Token — nunca UCT diretamente.

04
BFF → LaaS API AGREGAÇÃO
BFF agrega múltiplas APIs com o Service Token
Para montar uma resposta completa para o MFE, o BFF pode precisar chamar mais de uma API. Ele faz isso em paralelo e agrega o resultado — economizando round-trips do browser.
Node.js · BFF · aggregator.ts
// Paralelo — economiza tempo de resposta
const [limit, conditions, riskProfile] = await Promise.all([

  // API 1: limite de crédito disponível para o usuário X
  lendingApi.get(`/v1/credit-limit/${ctx.sub}`,
    { headers: { Authorization: `Bearer ${serviceToken}` } }),

  // API 2: condições comerciais do parceiro para capital de giro
  lendingApi.get(`/v1/conditions/capital-giro/${ctx.partner_id}`,
    { headers: { Authorization: `Bearer ${serviceToken}` } }),

  // API 3: score de risco do usuário (Squad 2 — Motor de Risco)
  riskApi.get(`/v1/risk-profile/${ctx.sub}`,
    { headers: { Authorization: `Bearer ${serviceToken}` } }),
]);

// Agrega e filtra conforme contrato do parceiro
return buildSimulationResponse(limit, conditions, riskProfile, ctx);
05
BFF · filtros REGRAS DE PARCEIRO
BFF aplica filtros específicos do parceiro
Antes de devolver ao MFE, o BFF aplica regras do parceiro — campos que não foram contratados são removidos, limites são ajustados, campos obrigatórios são validados. Essa lógica nunca fica no MFE nem nas APIs de plataforma.
Node.js · BFF · partnerFilter.ts
function applyPartnerFilters(data, partnerConfig) {

  // Remove campos não contratados pelo parceiro
  if (!partnerConfig.features.showRiskScore)
    delete data.riskScore;

  // Aplica teto de limite definido em contrato
  if (data.limit > partnerConfig.maxCreditLimit)
    data.limit = partnerConfig.maxCreditLimit;

  // Ajusta taxas conforme spread acordado com o parceiro
  data.rate = data.baseRate + partnerConfig.spreadBps / 10000;

  return data;
}
06
BFF → MFE RESPOSTA
BFF devolve resposta agregada e filtrada ao MFE
O MFE recebe uma resposta única, limpa, já filtrada pelo parceiro. Ele não sabe quantas APIs foram chamadas por baixo. O MFE só precisa renderizar os dados.
JSON · BFF → MFE · resposta
{
  "simulation": {
    "availableLimit": 50000,         ← de /credit-limit
    "monthlyRate":    0.012,           ← de /conditions + spread parceiro
    "maxTerm":        24,              ← de /conditions
    "gracePeriod":    30,              ← de /conditions
    "approvalChance": "high"           ← de /risk-profile (filtrado)
  },
  "meta": {
    "correlationId": "req_9a1c-f3b2", ← rastreabilidade ponta a ponta
    "cachedAt":      "2026-05-27T10:31:00Z"
  }
}

⏱ Tempo total: ~180ms
   (3 APIs em paralelo · filtros · serialização)
✓ O que o MFE nunca vê

Service Token · credenciais mTLS · score numérico bruto (se parceiro não contratou) · campos de risco interno · estrutura interna das APIs de plataforma. O BFF é o guardião dessa fronteira.

O Que o BFF Faz — e o Que Não Faz

O BFF é intencionalmente pequeno e focado. Três responsabilidades fixas, nada mais.

🔐
Segurança de Fronteira
RESPONSABILIDADE 1
Valida o UCT (assinatura, expiração, blacklist, partner match). Faz a troca por Service Token via mTLS. As APIs de plataforma nunca recebem UCT diretamente — só Service Token emitido pelo BFF.
Checklist de segurança
✓ Verifica assinatura JWT (JWKS)
✓ Checa jti no Redis blacklist
✓ Valida partner_id do token vs header
✓ Verifica scope para o endpoint chamado
✓ mTLS para chamadas saindo do BFF
✗ Nunca repassa UCT para APIs internas
🧩
Agregação de Dados
RESPONSABILIDADE 2
Chama múltiplas APIs em paralelo e devolve uma resposta única e coerente para o MFE. O MFE nunca faz N chamadas para montar uma tela — sempre uma.
Padrão de agregação
MFE faz 1 chamada → BFF
BFF faz N chamadas em paralelo:
  → /credit-limit      (LaaS)
  → /conditions        (LaaS)
  → /risk-profile      (Motor Risco)
BFF agrega → 1 resposta ao MFE

Resultado: tela mais rápida
Menos dados sensíveis no browser
⚙️
Filtros de Parceiro
RESPONSABILIDADE 3
Aplica as regras comerciais do parceiro: teto de limite, spread de taxa, campos habilitados, produtos contratados. Essa camada protege tanto o Bradesco (regras de negócio) quanto o parceiro (SLA comercial).
Exemplos de filtros
partner: banco-xpto
maxCreditLimit: 100_000   ← teto contratual
spreadBps:      85         ← spread de taxa (0,85%)
showRiskScore:  false       ← não contratou
maxTerm:        18          ← prazo máximo permitido
gracePeriod:    30          ← carência padrão XPTO
📊
Observabilidade
RESPONSABILIDADE 4
Emite traces distribuídos com o correlationId propagado do MFE. Qualquer incidente pode ser rastreado do clique do usuário até a API de plataforma. Métricas por parceiro, por produto e por endpoint.
Dados emitidos por requisição
correlationId  ← propagado do MFE
partnerId      ← do UCT validado
productId      ← capital-giro
endpoint       ← /simulate
durationMs     ← tempo total
upstreamCalls  ← [ {api, ms} ]
statusCode     ← 200 / 4xx / 5xx
🚫
O Que o BFF NÃO Faz
LIMITES INTENCIONAIS
O BFF tem um escopo deliberadamente estreito. Fora disso, é problema de outra camada.
Fora do escopo do BFF
✗ Regras de negócio de crédito (→ LaaS)
✗ Motor de score/risco      (→ Squad 2)
✗ Gestão de contratos       (→ LaaS)
✗ Emissão de tokens         (→ Core Platform)
✗ Multi-tenant IAM          (→ Core Platform)
✗ Kafka / eventos           (→ Core Platform)
✗ Lógica de UI              (→ MFE)
🚀
Stack & Deploy
TECNOLOGIA
Um BFF por produto de crédito. Stateless, leve, deployado no Azure AKS junto com as APIs de plataforma. Escala independente do MFE e do Core Platform.
Tech stack do BFF
runtime:   Node.js 20 LTS
framework: Fastify  ← baixa latência
http:      Axios + retry/circuit-breaker
auth:      jose (JWT) + redis (blacklist)
tls:       node-tls (mTLS client cert)
deploy:    Docker → AKS · Azure
scaling:   HPA (CPU + RPS por pod)
observ:    OpenTelemetry → Azure Monitor

Sem BFF vs Com BFF

Por que não conectar o MFE diretamente nas APIs de plataforma? Três problemas concretos que aparecem imediatamente.

✗ Sem BFF — MFE chama API diretamente
⚠️Browser teria que fazer mTLS — impossível de forma segura no frontend
⚠️UCT enviado diretamente para APIs bancárias — superfície de ataque maior
⚠️MFE faz 3 chamadas separadas para montar uma tela — 3x o round-trip
⚠️Filtros de parceiro ficam no MFE — lógica de negócio no frontend
⚠️Cada MFE reimplementa validação de token — duplicação e inconsistência
⚠️Troca de regra comercial de parceiro exige redesploy do MFE
⚠️Dados brutos das APIs expostos no tráfego do browser
✓ Com BFF — MFE fala só com o BFF
✅mTLS fica no BFF — certificados server-side, sem expor ao browser
✅UCT validado e trocado no BFF — APIs internas recebem Service Token
✅MFE faz 1 chamada — BFF agrega N APIs em paralelo
✅Filtros de parceiro no BFF — lógica de negócio server-side
✅Validação de token centralizada no BFF — um só lugar para atualizar
✅Regra comercial muda no BFF — MFE não precisa de redesploy
✅MFE recebe só o que precisa renderizar — dados sensíveis ficam server-side
⚠ Quando o BFF vira problema

BFF pode virar gargalo se a Squad 4 colocar regras de negócio pesadas nele. A linha é clara: orquestração e filtro = BFF. Regra de negócio de crédito = LaaS/CaaS. Se o BFF começa a calcular parcelas ou decidir aprovação, ele saiu do seu papel.
